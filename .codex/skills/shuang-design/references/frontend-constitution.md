# 前端设计系统 Constitution

用于把 `DESIGN.md` 与 Stitch 视觉参考转成 `.specify/memory/constitution.md` 的前端设计系统约束。

## 第一步：读取上下文

先读取：

1. `DESIGN.md`
   - 设计系统全貌。
   - 设计原则。
   - tokens。
   - 组件基底。
   - 暗色 / 亮色模式。
   - 响应式规则。
   - 页面和组件规范。

2. `specs/prd.md`
   - 业务背景。
   - 目标用户。
   - 目标市场。
   - 用户场景。
   - 产品边界。
   - 市场惯例。

3. `design-reference/stitch-export/`
   - 任选 2-3 个代表页面。
   - 优先首页、核心工作台、详情或编辑页。
   - 如果页面不足 2 个，如实说明。

如果文件缺失或路径不一致，先查找最接近候选文件，并在回复里说明实际读取了哪些文件。

## 第二步：抽取 constitution 级原则

更新或追加：

```md
## 前端设计系统

### 原则一：...

### 原则二：...

### Open Questions
```

每条原则必须包含：

- 原则名。
- 为什么不可违反。
- 执行依据：引用具体文件和段落 / 小节。
- 违反示例：说明哪类实现会违反原则，但不写具体 token 值。

原则是抽象方向，不是具体实现值。

## 必须覆盖的方向

1. 视觉规范的真理来源。
2. 视觉参考样本的使用方式。
3. 目标市场 / 用户群的不可妥协约定。
4. 组件库基底。
5. 信息密度 / 设计哲学。
6. 多语言策略。
7. 暗色 / 亮色模式。

## 硬约束

- 不复制任何 hex 值。
- 不复制任何 token 名。
- 不复制任何具体组件名。
- 不复制任何字体名。
- 不把 `DESIGN.md` 缩写成 constitution。
- 不替用户做产品决策。
- 没明说的事写进 Open Questions。
- 不覆盖 constitution 里已有的非前端原则。
- 保持 constitution 原有格式、语气和层级。

## 完成报告

回复：

1. 改了哪些文件。
2. 实际读取了哪些上下文文件。
3. 6-7 条原则分别引用哪些文件和段落 / 小节。
4. 是否存在 Open Questions。
5. 是否遵守“不复制 hex 值、token 名、组件名、字体名”。
