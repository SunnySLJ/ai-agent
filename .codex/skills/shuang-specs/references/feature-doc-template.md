# Feature 文档 4 步模板

用于从 `specs/prd.md` 的 Must-have 功能生成逐 feature 文档。

## Step 0：拆分与排序

只拆 PRD 中标记为 Must-have 的功能。先输出：

```md
| # | feature 名 | 来自 PRD 哪一节 | 依赖哪些 feature | 文档输出目录 |
|---|---|---|---|---|
```

目录格式：

```text
specs/00X-<feature-slug>/
```

按依赖关系排序，等待用户确认后再开始生成文档。

## Step 1：specify -> spec.md

要求：

- 只写 What & Why。
- 禁止写数据库选型、框架选型、库选型等技术细节。
- 必含功能边界、MVP 约束、运行环境、业务流程。
- 从 PRD 对应章节抽输入、输出、AC、边界条件。
- AC 使用 Given/When/Then。
- 长度参考 1500-2000 词；如果项目实际更适合中文，可用等价中文篇幅。

## Step 2：clarify -> 更新 spec.md

必须自动扫描刚生成的 `spec.md`，重点查：

1. MVP 边界：哪些不做是否写明。
2. API 边界：限频、失败、超时、配额。
3. 数据边界：空值、异常值、极端规模。
4. 集成边界：与现有系统交互的假设。

每个 clarify 问题要追到 `spec.md` 的具体段落更新。

如果用户也不确定，停下来问，不替用户决策。

## Step 3：plan -> plan.md

必须包含：

1. 项目文件结构：路径 + 每个文件核心职责。
2. 数据流向：Mermaid 图。
3. 依赖清单：语言版本、第三方库版本。
4. 与现有系统的集成点：复用哪些已有模块，哪些新建。
5. 风险点清单：技术风险 + 缓解方案。

复用优先：能复用 PRD 架构基线或现有模块的，绝不新建。

## Step 4：tasks -> tasks.md

要求：

- 12-18 条任务。
- 单一职责。
- 可单独测试。
- 入参出参明确。
- 2-5 分钟可完成。
- 标注并行组。
- 每条任务标注 `[FR-X 来源] [依赖任务] [出参验证方式]`。

## 硬约束

1. `/specify` 阶段不写技术选型。
2. `/clarify` 必跑。
3. `/plan` 缺 5 项要返工。
4. `/tasks` 控制在 12-18 条。
5. 目录必须遵守 `specs/00X-<feature-slug>/`。
6. 每个 feature 跑完 4 步后暂停给用户审。

## 遇到问题

- PRD 缺 AC、边界、数据样例：停下来问。
- clarify 问题用户也不确定：停下来问。
- 集成点不清楚：让用户提供现有系统代码路径或文档，不要猜。

## 输出节奏

一个 feature 跑完 4 步 -> 暂停 -> 用户确认 -> 下一个 feature。
