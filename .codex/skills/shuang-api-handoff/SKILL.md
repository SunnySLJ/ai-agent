# shuang-api-handoff · 接口联调交付

把一个已经开发或接近完成的后端接口，整理成前端可以直接联调的交付包：真实代码链路、关键注释、流程图、字段说明、风险清单、验证记录和 Apifox 可导入 JSON。

## 绝对约束

1. **真实代码优先**：先读 Controller、Param、VO、Service、Mapper/XML、配置、Feign/第三方调用和数据库变更，再写文档；不知道的地方标“待确认”，不要猜。
2. **只交付前端需要调用的接口**：内部回调、定时任务、供应商回调、管理端调试接口默认排除；除非前端真的需要调用，才写进文档和 Apifox。
3. **密钥不落盘**：Apifox token、服务器密码、签名密钥只从环境变量或用户会话读取，禁止写入 skill、文档、JSON、日志和最终回复。
4. **注释只补关键处**：补 model 类/字段、对外 Service 方法、复杂编排、事务边界、状态推进、第三方调用、兼容兜底；不写逐行解释，不顺手格式化无关代码。
5. **行号最后刷新**：代码注释或结构调整后，再用带行号读取文件，文档里的方法和行号必须来自最终文件。

## 工作流

### 0. 收敛范围

先确认本轮只处理哪些接口和功能。能从仓库判断就直接判断；影响交付边界时再问用户。

- 读取当前工作区状态，识别未提交改动和新增文件。
- 找到本轮接口入口：路由路径、HTTP 方法、Controller 方法、Param、VO/Result。
- 明确不触碰范围：非本功能接口、非前端接口、历史无关重构、密钥配置。
- 若目标仓库有项目规范，按该仓库的文档目录、注释规范和验证方式执行。

### 1. 建接口清单

按“前端能调用什么”建清单，不按后端文件列表堆材料。

| 项 | 必查内容 |
|---|---|
| 入口 | Controller 路径、HTTP 方法、方法名、鉴权要求 |
| 入参 | Param 字段、校验注解、默认值、必填条件、枚举、金额/时间单位 |
| 出参 | VO/Result 字段、兼容字段、前端拉起支付/跳转所需字段 |
| 业务链路 | Service 方法顺序、事务边界、状态推进、幂等键、异常分支 |
| 数据 | 表、实体、Mapper/XML、关键字段、唯一约束、落库副作用 |
| 外部依赖 | Feign/HTTP 客户端、配置项、供应商请求响应、超时和失败处理 |
| 环境 | 域名、网关前缀、测试环境配置、回调地址、需要前端传的 header |

### 2. 反推代码顺序

文档必须能回答“请求进来后代码按什么顺序跑”。按最终文件行号整理成表：

| 顺序 | 层 | 文件 | 方法/行号 | 职责 | 输入 | 输出/副作用 |
|---|---|---|---|---|---|---|
| 1 | Controller | `/abs/path/XxxController.java` | `createOrder:42` | 接参并传入用户上下文 | `XxxParam` | 调 Service |
| 2 | Service | `/abs/path/XxxServiceImpl.java` | `createOrder:88` | 校验、事务、状态推进 | DTO/Param | 订单/支付参数 |

行号要求：
- 使用最终文件的真实行号，推荐读取方式是 `nl -ba <file>` 或等价工具。
- 如果一个方法很长，标方法起始行即可；关键分支可额外标分支附近行。
- 不把未读过的文件写进表。

### 3. 补关键注释

先列出缺注释点，再最小改动补齐。常见补注释位置：

- `model/` 下类和字段：说明给前端的业务含义、单位、枚举值、必填条件。
- Controller 对外方法：说明接口用途和前端调用场景。
- Service public 方法：说明流程入口、事务边界、幂等/状态约束。
- 私有复杂方法：说明业务节点，例如开户提交、结算绑定、下单、支付参数转换、回调同步。
- Converter 的 `@AfterMapping`、默认值、旧字段兼容：说明为什么需要兜底。
- Feign/第三方调用封装：说明供应商动作，不暴露 token、签名、完整敏感请求。

不要补：
- getter/setter、普通赋值、显而易见的临时变量。
- 与本接口无关的历史类。
- 注释掉的废弃代码。

### 4. 做风险检查

输出一张“前端联调风险表”，按真实代码给结论：

| 风险 | 检查点 |
|---|---|
| 字段不全 | 前端必传字段是否都在 Param；字段名、类型、单位是否和旧前端兼容 |
| 出参不稳 | 前端需要的 `payData`、跳转 URL、订单号、状态字段是否固定 |
| 鉴权/租户 | 当前用户、租户、商户号是否从可信上下文取，不让前端伪造 |
| 幂等/重复提交 | 下单、开户、绑定、回调是否有业务唯一键或状态前置条件 |
| 金额/费率 | 是否使用 `BigDecimal`，单位是元/分，scale 和 rounding 是否明确 |
| 事务/补偿 | 落库、第三方调用、状态推进失败时是否可恢复 |
| Feign/环境 | 测试环境 URL、服务名、网关、超时、配置是否能打通 |
| 错误码 | 前端可识别的错误码和错误文案是否稳定 |
| 敏感信息 | token、签名、证件号、手机号、银行卡是否脱敏 |
| Apifox 契约 | 文档字段是否与最终 Param/VO 一致，隐藏接口是否已排除 |

### 5. 写联调文档

优先写到用户指定路径。若目标仓库有长期说明目录规则，遵守该规则；否则默认：

```text
docs/api-handoff/YYYYMMDD-<feature>.md
docs/apifox/<feature>-openapi.json
```

文档必须包含：

1. **联调摘要**：功能目标、前端入口、后端接口、测试环境、鉴权方式。
2. **接口清单**：只列前端调用接口，标出“不提供给前端”的排除项。
3. **主流程图**：用 Mermaid `flowchart TD`，覆盖前端动作、网关/鉴权、Controller、Service、DB、第三方、返回。
4. **时序图**：用 Mermaid `sequenceDiagram`，覆盖同步请求、异步回调、轮询/状态查询。
5. **状态图**：有订单/开户/支付状态时，用 Mermaid `stateDiagram-v2`。
6. **代码顺序表**：文件、方法、真实行号、职责、输入输出。
7. **请求字段表**：字段、类型、是否必填、来源、示例、注释、校验规则。
8. **响应字段表**：字段、类型、前端用途、示例、注释、兼容说明。
9. **错误码和失败分支**：错误码、触发条件、前端处理建议。
10. **数据和配置**：涉及表、关键字段、配置项、Feign/第三方地址。
11. **风险和待确认项**：只写真实检查后的问题，不写泛泛建议。
12. **验证记录**：编译、测试、JSON 校验、接口自测、Apifox 上传结果。

### 6. 生成 Apifox JSON

默认生成 Apifox 可导入的 OpenAPI 3.0 JSON。每个前端字段必须有 `description`，请求和响应都要有示例。

最低要求：
- `openapi: "3.0.3"`
- `info.title` 写功能名，`info.version` 写日期或接口版本。
- `servers` 写测试环境基础地址；未知时写占位并在文档标待确认。
- `paths` 只包含前端需要调用的接口。
- `components.schemas` 复用 Param/VO 命名，字段名与代码一致。
- 鉴权用 `components.securitySchemes` 表达，不把真实 token 写进 example。
- 用 `python3 -m json.tool <file>` 或等价方式校验 JSON。

上传 Apifox 规则：
- 只有用户明确要求“同步/上传 Apifox”时才上传。
- 使用 `APIFOX_ACCESS_TOKEN`、`APIFOX_PROJECT_ID` 等环境变量或用户当轮提供的信息；不要把 token 写进仓库文件。
- 上传前先查官方文档或仓库已有脚本，确认当前 Apifox API 端点和参数，不凭记忆硬调。
- 上传失败时保留本地 JSON，并在最终回复说明失败原因和下一步。

### 7. 交付前验证

按改动类型选择验证：

- 只写 skill/文档：跑 skill 结构校验、JSON 校验、检查链接/路径存在。
- 补了 Java 注释或轻微代码：跑项目要求的 Java 8 编译或至少说明为什么不能跑。
- 改了接口契约：尽量跑聚焦测试或本地接口请求；没有条件时给可执行复验步骤。
- 生成联调文档和 Apifox JSON：必须跑交付物检查：
  ```bash
  node scripts/api-handoff-artifact-check.mjs --doc <handoff.md> --openapi <openapi.json>
  ```
- 生成 Apifox JSON：必须确认 JSON 可解析，每个前端字段有 `description`，请求/响应有 example，且隐藏接口未进入 `paths`。

最终回复保持短而准：说明文档路径、Apifox JSON 路径、是否上传、做了哪些验证、还剩哪些待确认。

## 进化循环

每次真实使用后，如果发现前端反复问同类问题、Apifox 导入格式不稳定、字段注释遗漏、流程图不够细，先写：

```text
docs/skill-evolution/inbox/YYYY-MM-DD-<topic>.md
```

只有当经验满足长期化标准，再更新本 `SKILL.md` 或拆出 `references/`。不要把一次性业务事实写进 skill。
