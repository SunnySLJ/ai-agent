# shuang-next · 截图/原型图到 Next.js 直接开发

## 这个 skill 解决什么

当用户发截图、原型图、Figma 截图、竞品页面截图，并要求“按这个做”“翻译成 Next.js 代码”“直接开发好”“施工 spec”时，使用本 skill。

默认模式是 **B：直接开发模式**：先把截图转成足够明确的短施工 spec，再在当前 Next.js 项目中落地代码、验证构建。

目标不是抽象评价设计，而是把图片里的视觉、结构、文案、交互状态转译成当前项目可运行的 Next.js 页面或组件。

## HARD-GATE

- 当前真实前端项目是 `shuang-video`。
- 只能在当前项目中实现，不要新建独立 Next.js 项目。
- 优先改现有组件和现有 CSS；除非确实需要，不新增文件。
- 公开首页通常是 `shuang-video/src/components/HeroGenerator.tsx`。
- 登录后首页通常是 `shuang-video/src/components/LoggedInHome.tsx`。
- `/create` 页面通常是 `shuang-video/src/components/CreateWorkspace.tsx`。
- 全局样式主要在 `shuang-video/src/app/globals.css`。
- 不要把 Pollo 等竞品品牌文案带进最终 UI；默认替换为 Dream / 梦影 / DreamFilm。
- 参考图只取布局、层级、质感和交互状态，不按截图原始像素盲目放大。
- 弹层、下拉、chip、顶部栏默认保持当前项目的紧凑桌面比例。
- 用户明确要求“按照截图来”时，仍要适配当前项目比例；不要生成失控的大字号、大弹层。
- 如果图片中文字看不清，标记为 `[看不清]`，不要猜。
- 如果缺少实现必需信息，先问一个最关键的问题；不要边猜边写。
- UI 改完必须运行：

```bash
npm run typecheck --prefix ./shuang-video
npm run build --prefix ./shuang-video
```

## 什么时候使用

- 用户发截图并说“帮我做成这样”。
- 用户发原型图并要求“写成 Next.js”。
- 用户要求把产品图“翻译成 Codex 施工 spec”。
- 用户要求按图直接改现有页面。
- 用户对某个下拉、弹层、卡片、首页 hero、composer、sidebar、topbar 做视觉修正。

## 工作模式

### 默认：B · 直接开发模式

用户说“直接开发”“帮我弄好”“按截图做”时：

1. 识别目标页面和目标组件。
2. 输出 3-8 条短施工 spec，说明要改什么。
3. 读取目标组件和相关 CSS。
4. 小步修改 JSX / CSS。
5. 跑 typecheck 和 build。
6. 简短汇报改了哪些文件、验证结果。

### 可选：A · 施工 spec 模式

用户明确说“只要 spec”“先别写代码”“翻译成施工 spec”时，只输出 markdown spec，不改代码。

## 截图拆解流程

每次看图都按下面顺序拆，不要跳步。

### 1. 颜色

输出或内部确认这些颜色：

- 背景色：具体 6 位 hex。
- 卡片 / 面板色：具体 6 位 hex。
- 边框色：具体 6 位 hex。
- 文字三档：主文字、次文字、弱文字。
- Accent 色：最跳的主强调色。
- 状态色：绿、红、蓝、紫。

如果无法从截图精确取色，使用“视觉估算 hex”，并优先贴近项目已有 CSS token。

### 2. 字体 / 圆角 / 间距 / 阴影

必须具体到 px：

- 字号档：标题 / 正文 / 小字 / badge。
- 圆角档：大卡片 / 中按钮 / 小 badge。
- 间距系统：按 4 / 8 / 12 / 16 / 20 / 24 / 32 推断。
- 阴影：无、弱、中、强；给出大致 box-shadow。

### 3. 布局拆成网格

确认：

- 是否有 sidebar；宽度多少；分几组；每组内容。
- 是否有 topbar；高度多少；左中右各是什么。
- 主区几列几行；每个 cell 放什么模块。
- 弹层 / 下拉锚定在哪个按钮上方或下方。

复杂页面必须写 ASCII 草图；局部控件可以用一两行描述。

### 4. 文字 / 图标 / emoji 全抄

- 所有清晰可见中文、英文、数字、badge、状态点、emoji 都要转写。
- 看不清的写 `[看不清]`。
- 竞品品牌词进入实现前要替换成 Dream / 梦影。

### 5. mock 适配真实项目

截图里的数字和项目真实状态冲突时，以项目真实状态为准。

例：

- 截图是 `Agent 8 个`，项目只有 1 个，则实现写 1 个。
- 截图模型叫 `Pollo 2.5`，本项目写 `Dream 2.5` 或 `梦影 2.5`。
- 截图品牌是竞品，最终 UI 使用 Dream / 梦影。

### 6. 唯一真功能

每次实现前明确“唯一真功能”：

- 如果用户只要静态视觉复刻：写“本次无真功能，全部为展示 mock”。
- 如果有真实交互：只实现一个核心真实功能，其它视觉为 mock。
- 不要在一次截图复刻里扩展多个新功能。

## 施工 spec 模板

当需要输出 spec 时，使用这个结构：

```markdown
# <页面/组件名> 施工 spec

## 整体目标
一句话说明要把什么图转成什么页面/组件。

## HARD-GATE
- 技术栈、目录、禁止事项、唯一真功能。

## 设计语言
- 颜色：背景 / 卡片 / 边框 / 文本 / accent / 状态色。
- 字体：标题 / 正文 / 小字 / badge。
- 圆角：卡片 / 按钮 / badge。
- 间距：4 / 8 / 12 / 16 / 20 / 24 / 32。
- 阴影：具体 box-shadow。

## 布局结构
ASCII 草图 + 文字说明。

## 模块内容
逐模块列出文案、数字、图标、badge、状态。

## 唯一真功能
API / 数据流 / 状态流；如果没有真实功能，明确写无。

## 文件结构建议
- 文件 A：改什么，大约多少行。
- 文件 B：改什么，大约多少行。

## 验证方式
- typecheck
- build
- 需要浏览器确认的关键路径
```

## 直接开发执行细则

1. 先简短说明目标页面：公开首页 / 登录后首页 / `/create` / 其它。
2. 用 Read 读取目标组件和 CSS 段。
3. 优先做局部 CSS 覆盖，不盲改全局 `.chip` / `textarea`。
4. 小步 edit，避免一次性大范围重写。
5. 不新增复杂抽象，不引入新依赖。
6. UI 组件状态至少包含：默认、hover/active、展开态、禁用态（如果截图中有）。
7. 弹层要检查：层级、overflow、锚点位置、是否被父容器裁切。
8. 写完跑 typecheck/build。
9. 如果 dev server 正在跑，提醒用户刷新页面；不要声称已像素验收，除非实际浏览器检查过。

## 禁止词和输出要求

施工 spec 中禁止使用这些模糊词作为最终要求：

- 差不多
- 类似
- 合理
- 大概这样
- 按感觉

必须给具体值：色号、px、文案、文件路径、验证命令。

## 项目品牌规则

- UI 文案优先中文。
- 品牌默认用 梦影 / Dream / DreamFilm。
- 模型名默认用 Dream 2.5、Dream 2.0、Dream 3.0、Dream 1.6，或现有代码中的梦影模型命名。
- 不要在最终界面保留 Pollo 品牌，除非用户明确要求保留竞品名作对照。

## 完成标准

- 目标截图中的主要布局、层级、文案、状态已落地。
- 尺寸适配当前项目紧凑比例。
- 没有竞品品牌误留。
- `npm run typecheck --prefix ./shuang-video` 通过。
- `npm run build --prefix ./shuang-video` 通过。
